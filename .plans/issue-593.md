# Implementation Plan — Issue #593

## Goal

Add a `SmartNotificationModel` SQLAlchemy ORM class for a new `smart_notifications` table with multi-tenant fields, three PostgreSQL enum columns (priority, channel, timing), and a JSON-typed recipient_filter column. Register the model via the existing auto-discovery mechanism and run an autogenerate migration.

## Affected Files

- `src/db/models/smart_notification.py` — new file with `SmartNotificationModel` class
- `alembic/versions/<new_revision>.py` — autogenerated migration file (created by Alembic)
- `alembic/versions/3ea69d66514e_sync_models_with_db.py` — may need review after autogenerate to confirm no phantom diffs

## Implementation Steps

1. **Create `src/db/models/smart_notification.py`**
   - Define three Python `enum.IntEnum` classes: `Priority` (urgent=0, normal=1, low=2), `Channel` (email=0, sms=1, push=2, in_app=3), `Timing` (immediate=0, batch=1)
   - Import `Base` from `db.base`, `Mapped`, `mapped_column`, `Integer`, `String`, `DateTime`, `JSON`/`JSONB`, `func` from SQLAlchemy
   - Add `SmartNotificationModel` class with `__tablename__ = "smart_notifications"`
   - Columns: `id` (Integer, PK, autoincrement), `tenant_id` (Integer, nullable=False, index=True), `summarized_content` (String(1024), nullable=False), `priority` (imported enum, nullable=False, default=Priority.normal), `channel` (imported enum, nullable=False, default=Channel.email), `timing` (imported enum, nullable=False, default=Timing.immediate), `recipient_filter` (JSON, nullable=True), `created_at` (DateTime, server_default=func.now(), nullable=False)
   - Add `to_dict()` method returning all fields with `isoformat()` for `created_at`

2. **Run Alembic autogenerate migration**
   - Spin up a clean disposable `alembic_dev` database per CLAUDE.md instructions: `docker compose -f configs/docker-compose.test.yml up -d test-db`, drop and recreate `alembic_dev`, run `alembic upgrade head`
   - Run `alembic revision --autogenerate -m "add_smart_notifications_table"`
   - Review the generated `alembic/versions/<id>_add_smart_notifications_table.py` — check that the `upgrade()` creates the table with all columns, index on `tenant_id`, and enum/server defaults; verify `downgrade()` drops the table cleanly

3. **Verify migration applies and downgrades cleanly**
   - Run `alembic upgrade head` — should succeed with no errors
   - Run `alembic downgrade -1` — should succeed
   - Run `alembic upgrade head` again
   - Run a second `alembic revision --autogenerate -m "drift_check"` — the new file should contain only `pass` in both upgrade/downgrade, confirming no residual drift; delete that empty revision file if it is empty

4. **Confirm `alembic/env.py` will discover the new model**
   - `alembic/env.py` does `import db.models  # noqa: F401` which triggers `db/models/__init__.py`'s `pkgutil.iter_modules` auto-discovery — no changes needed to `alembic/env.py` or `db/base.py` for model registration

## Test Plan

- **Unit tests in `tests/unit/test_smart_notification.py`** (new file): Test `SmartNotificationModel` instantiation and `to_dict()` output; test enum values are correct; test default values are applied when fields are omitted
- **Integration tests in `tests/integration/`**: No integration test needed for this issue — the migration verification step covers database correctness. If a service is added for this model in a future issue, integration tests for CRUD operations would be added then

- `to_dict()` returns a dict with keys for all fields including ISO-formatted `created_at`
- `alembic revision --autogenerate` produces a non-empty migration with all columns and correct defaults
