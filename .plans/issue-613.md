# Implementation Plan — Issue #613

## Goal
Define two new SQLAlchemy ORM models — `ChatSessionModel` (live chat sessions) and `ChatMessageModel` (messages within a session) — and produce a reversible Alembic migration so the corresponding `chat_sessions` and `chat_messages` tables are created in PostgreSQL.

## Affected Files
- `src/db/models/chat_session.py` — new file; defines `ChatSessionModel`
- `src/db/models/chat_message.py` — new file; defines `ChatMessageModel`
- `alembic/versions/<id>_add_chat_sessions.py` — new migration file generated by `alembic revision --autogenerate`

## Implementation Steps

1. **Create `src/db/models/chat_session.py`** — Define `ChatSessionModel` inheriting from `db.base.Base`:
   - columns: `id` (PK, autoincrement), `tenant_id` (indexed), `user_id`, `title` (nullable String(200)), `created_at` (DateTimeTZ, server_default=func.now()), `updated_at` (DateTimeTZ, server_default+onupdate)
   - composite index `ix_chat_sessions_tenant_user` on (`tenant_id`, `user_id`)
   - relationship `messages: Mapped[list["ChatMessageModel"]]` with back_populates and cascade
   - `to_dict()` method serializing all fields
   - follow the exact same pattern as `AIConversationModel` in `ai_conversation.py`

2. **Create `src/db/models/chat_message.py`** — Define `ChatMessageModel` inheriting from `db.base.Base`:
   - columns: `id` (PK, autoincrement), `session_id` (FK to `chat_sessions.id`, cascade delete, indexed), `role` (String(20)), `content` (Text), `intent` (nullable String(100)), `created_at` (DateTimeTZ, server_default=func.now())
   - composite index `ix_chat_messages_tenant_session` on (`tenant_id`, `session_id`)
   - relationship `session: Mapped["ChatSessionModel"]` with back_populates
   - `to_dict()` method serializing all fields
   - follow the exact same pattern as `AIMessageModel` in `ai_conversation.py`

3. **Verify auto-import** — The models are auto-discovered by `db/models/__init__.py` via `pkgutil.iter_modules`, so no changes to `alembic/env.py` or `db/models/__init__.py` are needed. Confirm `import db.models` in `alembic/env.py` (already present) will pick up the two new modules.

4. **Spin up a clean `alembic_dev` database** (per CLAUDE.md):
   ```
   docker compose -f configs/docker-compose.test.yml up -d test-db
   docker exec configs-test-db-1 psql -U test_user -d postgres -c "DROP DATABASE IF EXISTS alembic_dev;"
   docker exec configs-test-db-1 psql -U test_user -d postgres -c "CREATE DATABASE alembic_dev;"
   ```

5. **Apply all prior migrations to bring the DB to head**:
   ```
   export PYTHONPATH=src
   export DATABASE_URL="postgresql+asyncpg://test_user:test_pass@localhost:5432/alembic_dev"
   alembic upgrade head
   ```

6. **Run autogenerate**:
   ```
   alembic revision --autogenerate -m "add_chat_sessions"
   ```
   This produces `alembic/versions/<id>_add_chat_sessions.py` with the CREATE TABLE and CREATE INDEX commands for both tables.

7. **Review and fix the generated migration** — autogenerate never gets it 100% right:
   - Confirm `down_revision` points to `c94d682d4b03` (the current head)
   - Verify `ondelete="CASCADE"` on the FK from `chat_messages.session_id` → `chat_sessions.id`
   - Confirm `server_default=sa.text('now()')` on timestamp columns and `sqlite_autoincrement=True` in table args
   - Ensure `upgrade()` and `downgrade()` are symmetric (drop indexes before drop table in downgrade; drop composite index first, then single-column index)

8. **Verify the migration applies cleanly and downgrades cleanly**:
   ```
   alembic upgrade head
   alembic downgrade -1
   alembic upgrade head
   ```

9. **Run drift check** — a second autogenerate should produce an empty migration (only `pass` in both functions); if so, delete it. If not, the initial migration was incomplete and must be corrected.

10. **Run lint and type check**:
    ```
    ruff check src/db/models/chat_session.py src/db/models/chat_message.py
    ruff format --check src/db/models/chat_session.py src/db/models/chat_message.py
    mypy src/db/models/chat_session.py src/db/models/chat_message.py
    ```

## Test Plan
- `tests/unit/test_chat_session_model.py` — new file; unit tests for `ChatSessionModel` construction and `to_dict()` output using `MagicMock`-based ORM fixtures (same pattern as `tests/unit/test_ai_service.py`)
- `tests/unit/test_chat_message_model.py` — new file; unit tests for `ChatMessageModel` construction and `to_dict()` output
- `tests/integration/test_chat_sessions_integration.py` — new file with `@pytest.mark.integration`; uses `db_schema`, `tenant_id`, `async_session` fixtures; inserts a `ChatSessionModel` and a `ChatMessageModel`, asserts both can be flushed/refreshed and that cascade-delete removes child messages when the session is deleted

## Acceptance Criteria
- `ChatSessionModel` and `ChatMessageModel` are importable via `from db.models import ChatSessionModel, ChatMessageModel`
- `alembic current` shows the new revision after `alembic upgrade head`
- `alembic downgrade -1` successfully drops both tables without error
- `alembic upgrade head` recreates both tables without error
- Unit tests in `tests/unit/` pass (`pytest tests/unit/test_chat_session_model.py tests/unit/test_chat_message_model.py -v`)
- Integration tests pass (`pytest tests/integration/test_chat_sessions_integration.py -v`)
- `ruff check` and `mypy` pass on the new model files with no errors
