# Implementation Plan — Issue #675

## Goal

Define `ImportJobModel` and `ExportJobModel` ORM models in `src/db/models/` and generate an Alembic migration so the tables exist in the database. No service or router logic is included — only the persistence layer.

## Affected Files

- `src/db/models/import_job.py` — new file: `ImportJobModel` with id, tenant_id, entity_type, file_path, status, total_rows, processed_rows, error_rows, created_at, updated_at
- `src/db/models/export_job.py` — new file: `ExportJobModel` with id, tenant_id, entity_type, fields (JSON), filters (JSON), file_path, status, expires_at, created_at, updated_at
- `alembic/versions/<hash>_add_import_export_jobs.py` — new migration file creating both tables, keyed to head revision `9d8e7f6a5b3c`

## Implementation Steps

1. **Create `src/db/models/import_job.py`** following the existing pattern (e.g. `task.py`, `workflow.py`):
   - `__tablename__ = "import_jobs"`
   - Fields: `id` (Integer PK), `tenant_id` (Integer, indexed), `entity_type` (String(100)), `file_path` (String(1000)), `status` (String(50), default `"pending"`), `total_rows` (Integer, default 0), `processed_rows` (Integer, default 0), `error_rows` (Integer, default 0), `created_at` and `updated_at` (DateTime with `server_default=func.now()`)
   - `to_dict()` method serialising all fields with isoformat for datetimes

2. **Create `src/db/models/export_job.py`** following the same pattern:
   - `__tablename__ = "export_jobs"`
   - Fields: `id` (Integer PK), `tenant_id` (Integer, indexed), `entity_type` (String(100)), `fields` (JSONB, default=dict), `filters` (JSONB, default=dict), `file_path` (String(1000)), `status` (String(50), default `"pending"`), `expires_at` (DateTime, nullable), `created_at` and `updated_at` (DateTime with server_default/onupdate)
   - `to_dict()` method

3. **Generate the Alembic migration** using the documented workflow:
   - Spin up `test-db` from `configs/docker-compose.test.yml`
   - Create/drop the `alembic_dev` database, run `alembic upgrade head`
   - Run `alembic revision --autogenerate -m "add import export jobs"` — this will diff `Base.metadata` (which now includes both new tables via the auto-import in `db.models.__init__.py`) against the `9d8e7f6a5b3c` head and produce the upgrade/downgrade SQL
   - Review the generated file: verify columns, types, defaults, indexes, and the downgrade drops both tables
   - Run upgrade → downgrade → upgrade against `alembic_dev` to confirm clean apply/revert
   - Run a second autogenerate to confirm no residual drift; delete the empty check migration if it contains only `pass`

4. **Verify `alembic/env.py` does not need changes** — `import db.models  # noqa: F401` auto-discovers all `.py` files in `src/db/models/` via `pkgutil.iter_modules`, so the new files are picked up without any import statements being added
